1) 데이터 타입 & 형변환
✔ Primitive vs Reference
Primitive
: byte → short → int → long → float → double
boolean, char 별도
Reference
: 클래스, 배열, 인터페이스
최상위: Object
✔ 형변환 (Casting)

Upcasting (자동)

A a = new B();

Downcasting (강제)

B b = (B) a;
※ 잘못하면 ClassCastException

-> 안전하게:

if (a instanceof B) {
    B b = (B) a;
}
2️) 다형성 (Polymorphism)

✔ 핵심
오버라이딩 (Override) → 재정의
오버로딩 (Overload) → 인자 다름
✔ 특징
부모 타입으로 선언 → 자식 객체 참조 가능
실행 시점에 결정됨 -> 동적 바인딩

3️) 동적 바인딩 vs 정적 바인딩
✔ Dynamic Binding (Virtual Invocation)
실행 시 실제 객체 기준으로 메서드 호출
-> 오버라이딩에서 발생
✔ Static Binding
컴파일 시 결정
static, final, private 메서드

4️) 캡슐화 & 기본 메서드
필드 → private
접근 → getter/setter
toString() → 객체 정보 출력

5️) 상속 & 구조
✔ 관계
is-a → 상속
has-a → 포함 (Association)
use-a → 의존 (Dependency)
✔ Shadow Effect
지역변수가 필드 가림
this.name

6️) final 키워드
class → 상속 불가 (ex: String)
변수 → 값 변경 불가
메서드 → 오버라이딩 불가

7️) static 핵심
✔ 특징
클래스 로딩 시 메모리 할당
객체 생성 없이 사용
✔ 용도
유틸 메서드
Math.random();

※ 주의
static 내부에서 non-static 접근 x
동적 바인딩 x
8️) 싱글턴 (Singleton)
✔ 목적

-> 객체를 딱 1개만 생성

✔ 구현 구조
class A {
    private static A instance = new A();
    private A() {}

    public static A getInstance() {
        return instance;
    }
}
※ 면접 포인트
멀티스레드에서 안전하지 않을 수 있음
해결: synchronized, LazyHolder 등
9️) 추상화 & Abstract
✔ 개념

-> “필요한 것만 추출”

✔ Abstract Class
객체 생성 x
공통 구조 제공
✔ Abstract Method
abstract void move();

-> 자식이 반드시 구현해야 함

10) 인터페이스 (* 핵심)
✔ 본질

-> 추상화 끝판왕
0> 기능 중심 설계

✔ 상속 vs 인터페이스
구분	기준
상속	"is-a" (본질)
인터페이스	"can-do" (기능)
✔ 역할

-> 객체 간 표준 (규격)

interface I {
    void m();
}
class B implements I {
    public void m() {}
}
✔ 장점
코드 일관성
교체 용이
I i = new B();
I i = new C();

-> 코드 변경 없음

✔ 특징
다중 구현 가능
변수 → public static final
메서드 → public abstract

✔ Java 8 이후
default 메서드 가능
(하지만 설계적으로 논쟁 있음)
※ 다이아몬드 문제
인터페이스 다중 구현 시 충돌 가능

1-3️ 예외 처리 (** 중요)
✔ Error vs Exception
구분	의미
Error	JVM 문제 (복구 불가)
Exception	코드 문제 (처리 가능)
✔ 처리 방식
1. 직접 처리
try {
} catch(Exception e) {
}
2. 위로 던지기
void m() throws Exception
✔ Runtime vs Checked
RuntimeException
개발자 실수 (버그)
ex) NullPointerException
String s = null;
s.length();

-> ※ 반드시 수정해야 함

Checked Exception
컴파일 시 강제 처리
※ 나쁜 코드
try {
} catch(Exception e) {}

-> x 예외 숨기기 (최악)

! 핵심 한줄 정리
다형성 = 부모 타입으로 다양한 객체 사용
인터페이스 = 기능 기준 설계 (유연성 핵심)
추상 클래스 = 구조 + 강제
static = 클래스 단위
싱글턴 = 객체 1개 보장
예외 = 숨기지 말고 해결
